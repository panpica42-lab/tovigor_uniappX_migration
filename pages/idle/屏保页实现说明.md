# 屏保页第一层实现说明

更新时间：2026-07-06

## 已实现

本次在 uni-app x 新项目中实现了屏保覆盖层 `components/idle/IdleOverlay.uvue`，并将首页 `pages/index/index` 恢复为 `pages.json` 的第一个页面。

应用启动后会先加载首页，首页加载过程中显示屏保覆盖层。用户点击任意位置后只隐藏覆盖层，不再从 `idle` 页面跳转到首页。这样首页在屏保展示期间已经完成组件创建、首屏渲染和资源加载，唤醒时接近“直接 show 出来”。

`pages/idle/idle.uvue` 仍然保留，但现在只是 `IdleOverlay` 的路由包装页，供后续“无操作进入屏保”时复用。

### 1. 全屏视频屏保

屏保页使用以下视频资源作为全屏背景：

```text
/static/icons/partTrainingActivity/startTraining/vedio-placeHolder.mp4
```

同时配置了 poster 兜底图：

```text
/static/icons/partTrainingActivity/startTraining/vedio-placeHolder-poster.jpg
```

视频配置保持和老项目一致的屏保行为：自动播放、循环播放、静音、铺满裁切，并关闭播放控件、进度条、手势播放、静音按钮、投屏按钮等交互控件。

uni-app x App 端不能只依赖 `autoplay`，当前实现额外给视频配置了稳定 `id="idleBgVideo"`，页面挂载后通过 `uni.createVideoContext('idleBgVideo')` 主动 `play()`。视频 `ended` 时也会手动 `seek(0)` 后重新播放，作为 `loop` 失效时的兜底。

屏保组件已移除常规 idle 调试日志，不再输出 `[idle] ...` 的加载提示。

### 2. 整屏点击唤醒

当前使用普通 `view.touch-mask` 作为视频的兄弟节点覆盖整个视频区域，让用户点击屏幕任意位置都触发唤醒。同时根节点、poster、video 本身也都绑定了 `click/tap` 兜底，避免 Android 原生 video 层级导致点击丢失。

注意：老项目 Vue 页面用了 `cover-view` 放在 `video` 内部，但当前 uni-app x Android 标准基座会提示：

```text
未找到 "cover-view" 组件，已自动当做 "view" 组件处理。
```

这说明当前运行基座没有识别 `cover-view` 组件。屏保页不需要为这个组件重新制作自定义基座，直接使用绝对定位的普通 `view` 覆盖层更符合当前 uni-app x 项目的实现规则，也可以消除这个 warning。

官方文档要点：

- uni-app x 的 Android App 页面是 `.uvue` 原生渲染，不是老 uni-app Vue 页面的 WebView 渲染。
- `cover-view` 在 App 端覆盖 video 的经验主要来自老 uni-app / nvue 场景，不能直接套到 app-uvue。
- 非 H5 端 video 是原生组件，层级和普通组件的覆盖关系需要实机验证。

参考：

```text
https://doc.dcloud.net.cn/uni-app-x/page.html
https://uniapp.dcloud.net.cn/component/cover-view.html
https://uniapp.dcloud.net.cn/component/video.html
```

在首页覆盖层模式下，点击后执行：

```text
showIdleOverlay.value = false
```

也就是只移除屏保层，直接露出已经加载好的首页。

在 `/pages/idle/idle` 路由包装页中，点击后仍执行：

```text
uni.redirectTo({ url: '/pages/index/index' })
```

如果 `redirectTo` 失败，则降级使用：

```text
uni.reLaunch({ url: '/pages/index/index' })
```

### 3. 防重复跳转

通过 `isNavigating` 标记防止连续点击重复触发跳转。跳转 complete 后延迟 500ms 重置状态。

## 目前未实现

第一层只实现了老项目屏保页本体，不包含全局空闲检测。以下内容还需要后续实现：

### 0. uvue 页面级预加载限制

`uni.preloadPage()` 官方平台差异说明中，App 端只支持 App-nvue 页面预加载，不支持 app-uvue 页面级预加载。

因此当前没有采用：

```text
uni.preloadPage({ url: '/pages/index/index' })
```

而是采用“首页先加载，屏保作为首页覆盖层”的方式，实现用户唤醒时直接显示已加载首页。

### 1. 无操作自动进入屏保

需要在 `App.uvue`、首页或全局事件层增加 idle timer，例如用户无操作 60 秒后自动跳转到：

```text
/pages/idle/idle
```

同时要明确哪些页面允许进入屏保，哪些页面禁止进入屏保。训练中、评估中、设备调试中通常不应自动进入屏保。

### 2. 全局交互监听

需要统一监听触摸、点击、按键等用户行为，用来重置空闲计时。uni-app x 中不同平台的事件可用性需要实机验证。

### 3. 更主动的资源预热

当前沿用老项目的隐藏 `image` 预加载方式。后续可以改为 `uni.getImageInfo()` 主动预热，并把首页、部位训练、教练头像等资源整理成共享预热模块，避免屏保页和首页维护两套列表。

### 4. 视频异常兜底

当前配置了 poster，但还没有增加 `@error` 处理。后续可以在视频播放失败时切换为静态背景图，避免出现黑屏。

### 5. 首页 ready 握手

后续可以让首页在首屏准备完成后打点或更新状态。屏保跳首页时可根据 `fromIdle=1` 之类参数跳过非必要动画，减少用户感知延迟。

### 6. 进入和退出策略

后续需要确定：

- 应用启动是否永远先进屏保
- 从首页闲置进入屏保时用 `redirectTo` 还是 `navigateTo`
- 从二级页面是否允许进入屏保
- 屏保返回首页后是否需要清理历史页面栈
