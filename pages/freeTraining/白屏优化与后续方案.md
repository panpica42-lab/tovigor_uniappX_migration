# 页面跳转白屏优化复盘与后续方案

## 1. 背景

本次问题集中在 uni-app X 迁移版本中，用户从首页进入二级页面时，会看到明显的页面间白屏。

重点分析路径：

```text
/pages/index/index
  -> /pages/partTraining/part-training
```

排查后确认：

- 主要慢点不是接口请求。
- 主要慢点不是图片压缩不足。
- 主要慢点集中在新页面创建原生视图、排版、渲染、提交首帧。
- 骨架屏对当前问题帮助有限，因为白屏主要发生在目标页首帧出现之前。

## 2. 优化前后日志对比

### 优化前

```text
进入页面: /pages/index/index
[
  { "创建dom元素个数": "134个", "耗时": "406ms" },
  { "排版": "1次", "耗时": "189ms" },
  { "渲染": "2次", "耗时": "762ms" },
  { "跳转页面到onReady总耗时": "1933ms" }
]
```

问题特征：

- DOM 创建 134 个，首屏视图偏重。
- 渲染发生 2 次，说明页面初始化过程中存在额外状态更新。
- onReady 总耗时接近 2 秒，用户体感白屏明显。

### 优化后，第一次进入目标页

```text
进入页面: /pages/partTraining/part-training?traceTs=1780556897678
[
  { "创建dom元素个数": "98个", "耗时": "317ms" },
  { "排版": "1次", "耗时": "98ms" },
  { "渲染": "1次", "耗时": "318ms" },
  { "跳转页面到onReady总耗时": "1208ms" }
]
```

### 优化后，再次进入相同目标页

```text
进入页面: /pages/partTraining/part-training?traceTs=1780556933911
[
  { "创建dom元素个数": "98个", "耗时": "91ms" },
  { "排版": "1次", "耗时": "37ms" },
  { "渲染": "1次", "耗时": "139ms" },
  { "跳转页面到onReady总耗时": "530ms" }
]
```

### 结果总结

```text
DOM 数量：134 -> 98，减少 36 个
渲染次数：2 -> 1
热态 onReady：1933ms -> 530ms
```

本次优化非常明确地降低了首屏创建压力，并消除了一个首屏阶段的额外渲染。

## 3. 本次实际改动

### 3.1 部位训练页首屏课程数量限制

文件：

```text
tovigor_yei_unIappX/pages/partTraining/part-training.uvue
```

改动：

- 增加 `visibleCourseLimit = ref(4)`。
- `visibleCourses` 不再直接返回全部课程，而是先返回前 4 个。
- `onReady` 后再把课程数量放开到全部。

目的：

- 首屏只创建 4 张课程卡。
- 避免进入页面时一次性创建全部 10 张课程卡。
- 降低原生视图创建、排版和渲染成本。

### 3.2 移除侧栏 mounted 阶段的反向更新

文件：

```text
tovigor_yei_unIappX/components/ui-box/CourseLibrarySidebar.uvue
```

原问题：

侧栏组件 `onMounted` 时会执行：

```ts
emit('selectCoach', selectedCoach.value)
```

父页面收到事件后会更新：

```ts
selectedCoachValues.value = [coachValue]
```

这会导致目标页首屏挂载后，子组件再反向通知父组件更新状态，从而触发额外渲染。

改动：

- 保留 `currentCoach.value = getSelectedCoach()`。
- 移除 mounted 阶段的 `emit('selectCoach')`。
- 用户真正选择教练时，仍然保留 `emit('selectCoach', coachData.value)`。

结果：

- 首屏渲染次数从 2 次降到 1 次。
- 教练选择功能不受影响。

### 3.3 AI 教练区默认收起

文件：

```text
tovigor_yei_unIappX/components/ui-box/CourseLibrarySidebar.uvue
```

改动：

```ts
const expandedCoach = ref(false)
```

目的：

- 首屏不渲染 AI 教练卡片列表。
- 用户点击“AI教练”筛选项后再渲染。
- 降低侧栏首屏 DOM 数量。

## 4. 为什么第一次进入仍然比第二次慢

优化后，第一次进入和第二次进入 DOM 数量都是 98 个，但耗时不同：

```text
第一次：
DOM 98 个，317ms
排版 98ms
渲染 318ms
onReady 1208ms

第二次：
DOM 98 个，91ms
排版 37ms
渲染 139ms
onReady 530ms
```

原因是第一次进入存在冷启动成本：

- 页面模块第一次加载。
- UTS/Vue 组件第一次初始化。
- 样式第一次解析。
- 原生视图类型第一次创建。
- 图片资源路径、解码、纹理缓存第一次命中。
- Android 运行时、JIT、渲染管线还没有进入热态。

所以，DOM 数量相同，不代表创建成本相同。

本次优化解决的是：

- 首屏工作量过大。
- 首屏阶段重复渲染。

但第一次进入页面时，仍然会有一部分框架和运行时冷启动成本。

## 5. 后续方案一：继续做每个页面的首屏减重

这是最稳妥、最通用的方向。

建议每个页面都按以下原则检查：

- 首屏只渲染用户立即能看到和必须操作的区域。
- 长列表首屏限制数量，例如 4、6、8 个，等 `onReady` 后再补齐。
- 默认隐藏的弹窗不要首屏挂载。
- 默认折叠区不要首屏展开。
- 避免子组件 `onMounted` 后立刻反向更新父组件。
- 避免页面初始化阶段连续多次改响应式数据。
- 重计算、筛选、排序、格式化尽量提前完成或延后到首帧之后。

适用范围：

- 所有普通页面。
- 所有二级页。
- 所有存在列表、筛选栏、弹窗、复杂组件的页面。

优点：

- 风险低。
- 不改变路由结构。
- 不影响返回栈。
- 优化收益稳定。

缺点：

- 不能完全消除第一次进入页面的冷启动成本。

## 6. 后续方案二：预热页面相关数据和资源

预热不是把页面提前打开，也不是把所有页面合并成一个页面。

更准确地说，预热是：

```text
在用户真正点击进入页面之前，提前让目标页面依赖的数据、资源、模块进入缓存或热态。
```

### 6.1 提前加载数据

适合内容：

- 课程数据。
- 教练配置。
- 筛选项配置。
- 动作数据。
- 训练流程配置。

做法示例：

```ts
import { partTrainingCourses } from '@/pages/partTraining/course-data.uts'
import { getAllCoaches } from '@/utils/coachManager.uts'

function warmupPartTrainingData() {
	const courseCount = partTrainingCourses.length
	const coachCount = getAllCoaches().length
	console.log('[Warmup] partTraining data ready courses=' + courseCount.toString() + ' coaches=' + coachCount.toString())
}
```

可以在首页 `onReady` 后调用，而不是 App 一启动就调用。

建议时机：

```text
首页 onReady 后 300ms-800ms
或者用户停留首页空闲时
```

优点：

- 风险低。
- 不创建页面 DOM。
- 不改变生命周期。

缺点：

- 对原生视图创建耗时帮助有限。
- 主要减少数据和模块首次访问成本。

### 6.2 提前缓存图片

适合内容：

- 目标页首屏课程封面。
- 高频入口卡片背景。
- 教练头像。

做法思路：

- 在首页空闲时，用隐藏或轻量方式触发图片资源加载。
- 或维护一个资源预热工具，按页面声明需要预热的图片列表。

注意：

- 不要一次预热大量图片。
- 不要在 App 启动第一帧预热。
- 优先预热核心链路的首屏图片。

建议策略：

```text
首页进入稳定后，只预热部位训练页前 4 张课程图 + 教练头像。
```

优点：

- 可降低首次图片解码、纹理准备成本。

缺点：

- 当前白屏主因不是图片，所以收益可能中等。
- 预热过多会增加内存和启动后负担。

### 6.3 提前 import 模块

适合内容：

- 目标页面使用的数据模块。
- 工具模块。
- 轻量配置模块。

目的：

- 让模块解析、初始化更早发生。
- 减少点击后才首次加载模块的成本。

注意：

- 不建议为了预热而 import 大量页面组件。
- 页面组件本身的生命周期和原生视图创建无法简单通过 import 完整预热。

推荐优先级：

```text
数据模块 > 配置模块 > 工具模块 > 页面组件
```

## 7. 后续方案三：核心入口使用页面内切换

这是收益最大，但风险也最大的方案。

不是把所有页面都改成一个页面，而是只针对极高频入口考虑。

适用场景：

- 首页到部位训练。
- 首页到自由训练入口。
- App 的主功能区。

思路：

```text
保留独立页面用于普通路由访问。
同时把核心页面主体抽成组件。
首页中可以直接切换显示该组件。
```

这样用户点击时不走 `navigateTo`，而是在当前页面内显示目标内容。

优点：

- 几乎可以消除页面路由切换白屏。
- 对最高频路径体验提升最大。

缺点：

- 会增加首页复杂度。
- 需要重新处理返回逻辑。
- 页面生命周期边界会变模糊。
- 内存占用可能上升。
- 迁移阶段不建议过早做大改。

建议：

```text
只在迁移稳定后，对最高频入口做专项评估。
不要把所有页面都合并进首页。
```

## 8. 推荐后续优先级

### 第一优先级：继续首屏减重

对每个慢页面做类似本次的检查：

- 首屏 DOM 数。
- 渲染次数。
- 是否 mounted 后二次更新。
- 是否默认挂载隐藏弹窗。
- 是否首屏一次性渲染长列表。

### 第二优先级：核心链路资源预热

先针对：

```text
首页 -> 部位训练
首页 -> 自由训练
部位训练 -> 课程详情
课程详情 -> 训练准备页
```

做轻量预热。

### 第三优先级：高频入口页面内切换

仅当首屏减重和预热仍无法满足体验目标时，再考虑。

## 9. 当前结论

这次优化证明：

```text
白屏时间和首屏 DOM 数量、渲染次数强相关。
```

本次从：

```text
134 个 DOM / 2 次渲染 / 1933ms onReady
```

优化到热态：

```text
98 个 DOM / 1 次渲染 / 530ms onReady
```

说明当前方向是有效的。

后续不应该简单把所有页面合并成一个巨型页面，而应该按页面重要程度分层处理：

```text
普通页面：首屏减重
核心页面：首屏减重 + 数据/资源预热
最高频入口：必要时再考虑页面内切换
```
